# Phase A – Attack Of The Clones (Exploration Notes)
## Patch Source

Patch URL: https://launchpadlibrarian.net/764050984/0001-Fix-invalid-nullptr-dereference-in-TagSection.keys.patch

Affected component: python/tag.cc (APT Python bindings)

## Nature of the Vulnerability

The patch fixes a potential NULL pointer dereference in the Python bindings of APT.

Specifically, PyString_FromStringAndSize() may return nullptr when invalid Unicode input is encountered.
In the original code, the returned object was used without checking for failure, which could lead to a crash.
The patch introduces an explicit NULL check and aborts cleanly when object creation fails.

## Extracted Code Fragments (from patch)

The following added lines were extracted automatically from the patch using a small Python script:

PyObject *Obj = PyString_FromStringAndSize(Start, End-Start);

if (Obj == nullptr)

return Py_DECREF(List), nullptr;

PyList_Append(List, Obj);

These lines contain stable API identifiers and control-flow logic that are suitable candidates for deriving fuzzy code signatures.

## Initial Observations

API and function names such as PyString_FromStringAndSize and PyList_Append are stable across packages.

The NULL-check pattern is simple and semantically meaningful.

Test-related changes (e.g. in tests/test_tagfile.py) should be excluded from signature generation.

This patch is well suited for experimenting with a patch → signature → codesearch workflow.

## Candidate Fuzzy Signatures

The following RE2-compatible fuzzy signatures were derived manually from the extracted lines:

PyString_FromStringAndSize\s*\(

if\s*\(\s*Obj\s*==\s*nullptr\s*\)

These signatures are intentionally loose and avoid advanced regex features not supported by RE2.

## CodeSearch Experiments

The signature PyString_FromStringAndSize\s*\( was tested against codesearch.debian.net.

Results included matches in unrelated packages such as:

vim/src/if_python.c

libwebp/swig/libwebp_python_wrap.c

libwebp/swig/libwebp.swig

These matches include both direct call sites and autogenerated wrapper code, illustrating how similar logic can appear across the archive due to manual duplication, vendoring, or code generation.

The NULL-check–based signature produced fewer matches and appeared more precise, suggesting that combining API-based and control-flow–based signatures could help balance recall and precision.

## Key Result

This small experiment confirms that patch-derived, RE2-compatible fuzzy signatures can identify semantically similar code patterns across unrelated Debian packages.

While the results are exploratory and include some noise (e.g. macro definitions or generated code), they demonstrate that security patches can serve as effective starting points for detecting cloned code at distribution scale, provided that appropriate filtering and refinement are applied.
