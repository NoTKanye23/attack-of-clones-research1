# Experiment 1 – NULL Pointer Dereference (APT Python Bindings)

## Patch Source

Patch URL:
https://launchpadlibrarian.net/764050984/0001-Fix-invalid-nullptr-dereference-in-TagSection.keys.patch

Affected component:
`python/tag.cc` (APT Python bindings)

---

## Methodology

This experiment evaluates whether vulnerability patterns extracted from a security patch can be used to discover similar code fragments across the Debian archive.

The prototype pipeline performs the following steps:

1. Parse the security patch and extract vulnerable and fix fragments
2. Generate candidate vulnerability signatures from the patch
3. Filter noisy or trivial signatures
4. Rank signatures by structural importance
5. Query the Debian CodeSearch archive
6. Verify candidate matches using vulnerability and fix patterns

---

## Nature of the Vulnerability

The patch fixes a potential **NULL pointer dereference** in the Python bindings of APT.

The function `PyString_FromStringAndSize()` may return `nullptr` when invalid Unicode input is encountered.

In the original code, the returned object was used without checking for failure, which could lead to a crash.

The patch introduces an explicit **NULL check** and aborts safely if object creation fails.

---

## Extracted Code Fragments

Relevant code fragments extracted from the patch include:

```cpp
PyObject *Obj = PyString_FromStringAndSize(Start, End-Start);

if (Obj == nullptr)

return Py_DECREF(List), nullptr;

PyList_Append(List, Obj);
```

These fragments contain both **API calls** and **control-flow validation**, making them good candidates for signature extraction.

---

## Candidate Signatures

The prototype pipeline extracted several candidate signatures, including:

```
PyString_FromStringAndSize(
if (Obj == nullptr)
PyList_Append(
```

These signatures capture both:

* API-level operations
* control-flow validation patterns

---

## CodeSearch Experiments

The generated signatures were used to query the Debian CodeSearch archive:

https://codesearch.debian.net

Example signature used:

```
PyString_FromStringAndSize(
```

The search returned matches in packages such as:

* vim
* libwebp
* autogenerated SWIG bindings

These matches demonstrate how similar API usage patterns appear across unrelated packages.

---

## Results

Matches found: **4**

Manual classification:

* Estimated false positives: **3**
* Estimated true positives: **1**

Most matches corresponded to unrelated uses of the same API function.

Only one match exhibited a similar **NULL-check validation pattern**.

---

## Observation

API-based signatures such as `PyString_FromStringAndSize` generate cross-package matches because API calls are often reused across projects.

However, API signatures alone produce significant noise.

Control-flow signatures such as:

```
if (Obj == nullptr)
```

appear to be more precise indicators of vulnerability patterns.

Combining **API identifiers** with **control-flow patterns** appears to provide the best balance between recall and precision.

---

## Conclusion

This experiment demonstrates that **patch-derived signatures can identify semantically similar code fragments across the Debian archive**.

However, single-token API signatures often produce noisy results.

Improved precision can be achieved by combining multiple signal types, including:

* API calls
* control-flow checks
* structural context patterns

This confirms the feasibility of the **patch → signature → archive search workflow** for detecting potential vulnerability clones.
